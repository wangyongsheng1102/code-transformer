# Struts2 → Spring Boot 代码转换工具（JavaParser）需求一览

## 1. 项目目标

- **目标**：基于 **JavaParser** 实现一个后端代码转换引擎，将传统 **Struts2** 项目的后端 Java 代码，半自动迁移为 **Spring Boot + Spring MVC** 工程。
- **范围**：
  - 只处理 **后端 Java 代码 + Struts 配置**（`struts.xml` 等）。
  - 暂不处理 JSP/标签库（Struts 标签、OGNL 表达式等前端视图层）。
- **预期效果**：
  - 自动完成大部分 **机械性、模式化** 的迁移（类/方法结构、包名、注解、import 等）。
  - 为复杂逻辑生成尽可能合理的骨架与 TODO 注释，便于人工二次修改。
  - 最终生成一个可编译的 **Spring Boot 工程骨架**。

---

## 2. 总体架构与流程

### 2.1 核心模块

- **项目扫描模块**
  - 扫描源工程：`src/main/java`，`src/main/resources`。
  - 找出：
    - Struts **Action 类**（`extends ActionSupport`、`implements Action`，或在 `struts.xml` 中被 `<action>` 引用）。
    - Struts 自定义拦截器、验证器等类。
    - `struts.xml`、`struts.properties` 等配置文件。

- **解析与建模模块**
  - 使用 **JavaParser** 解析所有 `.java` 文件为 AST。
  - 解析 `struts.xml`：
    - 建立 `namespace + action name + method` → `Action 类/方法` 的映射。
    - 建立 `result name` → 视图路径的映射。
    - 建立拦截器栈配置的简单模型（名称、顺序等）。

- **规则引擎模块（XML 配置驱动）**
  - 使用 **XML 文件**定义规则，且规则本身**分为“基础规则层”和“特定规则层”**，描述：
    - 包名转换规则（packageRules）。
    - import 增删改规则（importRules）。
    - 类级别转换（Action → Controller、Interceptor → HandlerInterceptor 等）（classRules）。
    - 字段/注入方式转换（fieldRules）。
    - 方法级别转换（方法签名、注解、返回值等）（methodRules）。
    - 参数与 Servlet API 使用模式的转换（paramRules）。
    - 路由和 result 映射规则（routeRules）。
  - 每条规则都支持：
    - **规则类型**：基础（BASE）/ 特定（PROJECT/PROFILE）；
    - **启用开关**：enabled；
    - **优先级**：priority，用于解决多条规则同时命中时的覆盖/合并策略。
  - 负责将规则与 AST 进行匹配，并按“基础规则先执行，再叠加特定规则”的策略生成对应的 AST 变换操作。

- **AST 变换与代码生成模块**
  - 基于规则，对 AST 做增删改：
    - 包声明（package）。
    - import。
    - 类继承/实现结构。
    - 注解（类/字段/方法）。
    - 方法签名（名称、参数列表、返回类型）。
    - 方法体中典型模式的替换（如 `ServletActionContext.getRequest()` 等）。
  - 将修改后的 AST **回写为新的 Java 源文件**，输出到目标目录。
  - 生成基础 **Spring Boot 工程结构**（`pom.xml`/`build.gradle`、启动类、基础配置等）。

- **日志与报告模块**
  - 记录每个文件、每个规则的应用情况（哪些类/方法被修改，做了什么修改）。
  - 可选输出：
    - 控制台日志。
    - 结构化日志（JSON）。
    - 简单 HTML/Markdown 报告，总结迁移情况与 TODO。

### 2.2 基本流程

1. 读取配置（基础规则 + 特定规则的 XML + 工程路径）。
2. 扫描并解析源工程（JavaParser + `struts.xml`）。
3. 根据规则分层与优先级，对 AST 进行一系列变换：
   - 先应用基础规则（base rules），实现通用迁移；
   - 再应用特定规则（project/profile rules），做项目个性化调整。
4. 输出变换后的代码到新工程目录。
5. 输出报告与日志。

### 2.3 规则分层：基础规则 + 特定规则

- **基础规则（Base Rules）**
  - 面向“所有 Struts2 → Spring Boot 项目”通用的迁移模式。
  - 例如：
    - 删除所有 `com.opensymphony.xwork2.*` import；
    - 将 `extends ActionSupport` 的类统一视为 Controller 候选；
    - 将 `ServletActionContext.getRequest()` 视为需要替换为方法参数的模式。
  - 可以提供一份“内置基础规则配置文件”，一般不随具体项目频繁修改。

- **特定规则（Project / Profile Rules）**
  - 针对某个具体项目或某类项目的定制规则。
  - 例如：
    - 某项目中所有 `com.foo.legacy.struts.action` 都要映射到 `com.foo.app.web.controller`；
    - 某些 Action 需要加特定的注解或额外的 Spring 组件。
  - 通过单独的配置文件或配置片段注入，覆盖/补充基础规则。

- **规则合并与优先级**
  - 规则引擎加载时，支持同时加载：
    - 一份全局基础配置（如 `base-rules.yml`）；
    - 一份或多份项目特定配置（如 `project-rules.yml`）。
  - 所有规则按以下顺序应用：
    1. 启用状态为 true 的 **基础规则**，按 priority 从小到大执行；
    2. 启用状态为 true 的 **特定规则**，按 priority 从小到大执行；
  - 当两条规则对同一 AST 元素产生冲突时，一般以：
    - **优先级高的规则**覆盖优先级低的规则；
    - 当优先级相同时，以 **特定规则覆盖基础规则**。

## 3. 规则维度、XML 配置结构与典型固定转换

### 3.1 XML 顶层结构设计（基础 + 特定规则）

- **文件拆分建议**
  - `base-rules.xml`：框架层面的通用基础规则，适用于所有 Struts2 → Spring Boot 迁移项目。
  - `project-rules.xml`：某个具体项目的特定规则，覆盖/补充基础规则。

- **XML 顶层结构示例**

```xml
<RuleSet name="BaseRules" type="BASE">
  <PackageRules>
    <PackageRule enabled="true" priority="100">
      <MatchPrefix>com.xxx.project.struts.action</MatchPrefix>
      <TargetPrefix>com.xxx.project.web.controller</TargetPrefix>
    </PackageRule>
  </PackageRules>

  <ImportRules>
    <ImportRule enabled="true" priority="100">
      <Match>com.opensymphony.xwork2.</Match>
      <Action>REMOVE</Action>
    </ImportRule>
  </ImportRules>

  <ClassRules>
    <ClassRule enabled="true" priority="100">
      <ExtendsClass>com.opensymphony.xwork2.ActionSupport</ExtendsClass>
      <RemoveExtends>true</RemoveExtends>
      <AddAnnotations>
        <Annotation>org.springframework.stereotype.Controller</Annotation>
      </AddAnnotations>
    </ClassRule>
  </ClassRules>

  <!-- 其他 MethodRules / FieldRules / ParamRules / RouteRules 可按类似结构扩展 -->
</RuleSet>
```

- **规则元信息字段**
  - `enabled`：是否启用该规则（默认 true）。
  - `priority`：执行优先级，数值越小越早执行（基础规则一般数值较小，项目规则可在同一范围内调整）。
  - 若区分基础/特定层级，可以：
    - 在文件级通过 `RuleSet` 的 `type="BASE|PROJECT"` 标注；
    - 或在每条规则上增加 `type` 属性，当前项目可先采用文件级区分，简化实现。

### 3.2 importRules：import 增删改规则（含新增）

- **删除 Struts/XWork 相关 import**
  - 删除：
    - `com.opensymphony.xwork2.ActionSupport`
    - `com.opensymphony.xwork2.Action`
    - `com.opensymphony.xwork2.ModelDriven`
    - `com.opensymphony.xwork2.interceptor.*`
    - `org.apache.struts2.ServletActionContext`
    - `org.apache.struts2.interceptor.*`
    - `org.apache.struts2.dispatcher.*`
- **替换为 Spring/Servlet API**
  - `org.apache.struts2.ServletActionContext` → 通过方法参数注入：
    - `jakarta.servlet.http.HttpServletRequest`
    - `jakarta.servlet.http.HttpServletResponse`
    - `jakarta.servlet.http.HttpSession`
  - 上传相关：
    - `org.apache.struts2.dispatcher.multipart.MultiPartRequestWrapper`
      → `org.springframework.web.multipart.MultipartFile` / `MultipartHttpServletRequest`
- **需求点**
  - 支持按“完整类名”或“包通配”（如 `com.opensymphony.xwork2.*`）匹配 import。
  - 行为类型（针对同一源码单元允许多条规则依次生效）：
    - **删除**：`removeImport`，移除 Struts 相关依赖；
    - **替换**：`replaceImport(旧 → 新)`，例如替换为 Spring MVC / Servlet API；
    - **新增**：`addImport`，为后续新增的方法/类提前补充所需依赖（如 `org.springframework.web.bind.annotation.*`）。
  - 需要支持在 XML 中显式声明“只新增不匹配删除”的规则，例如：

```xml
<ImportRule enabled="true" priority="200">
  <Match>org.springframework.web.bind.annotation.GetMapping</Match>
  <Action>ADD</Action>
</ImportRule>
```

### 3.3 packageRules：包名转换规则

- **典型映射**
  - `com.xxx.project.struts.action` → `com.xxx.project.web.controller`
  - `com.xxx.project.struts.interceptor` → `com.xxx.project.web.interceptor`
  - `com.xxx.project.struts.form` → `com.xxx.project.web.dto` / `web.request`
- **需求点**
  - 支持基于字符串前缀或模式的包名替换。
  - 修改 AST 中的 package 声明，并相应调整输出文件路径。

---

### 3.4 classRules：类级别转换（Action/Interceptor → Spring 组件）

#### 3.3.1 Action → Controller

- **识别条件（任一满足）**
  - `extends com.opensymphony.xwork2.ActionSupport`
  - `implements com.opensymphony.xwork2.Action`
  - 在 `struts.xml` 中被 `<action>` 配置引用。
- **转换动作**
  - 移除 Struts 基类/接口：
    - `removeExtends(ActionSupport)`
    - `removeImplements(Action)`
  - 增加 Spring MVC 注解：
    - `@Controller` 或 `@RestController`
    - 可选：根据 `struts.xml` 中 `namespace` 添加类级 `@RequestMapping("namespace")`
- **需求点**
  - 在规则中可配置：
    - 使用 `@Controller` 还是 `@RestController`。
    - 是否按 `namespace` 自动加 `@RequestMapping`。

#### 3.3.2 Interceptor → HandlerInterceptor

- **识别条件**
  - `implements com.opensymphony.xwork2.interceptor.Interceptor`
  - 或 `extends com.opensymphony.xwork2.interceptor.MethodFilterInterceptor`
- **转换动作**
  - 删除 Struts 拦截器接口/父类实现。
  - 增加实现：
    - `implements org.springframework.web.servlet.HandlerInterceptor`
  - 将 `intercept(ActionInvocation)` 中的逻辑迁移到 `preHandle`/`postHandle`/`afterCompletion`（可先全部放到 `preHandle` 并加 TODO 注释）。
- **需求点**
  - 生成 Spring 拦截器方法骨架。
  - 插入 TODO 注释提示人工检查逻辑映射。

---

### 3.5 fieldRules：字段与依赖注入规则

- **识别可注入字段**
  - 存在典型 setter 方法如 `setXxxService(XxxService service)`。
  - 字段类型在给定包（如 `*.service.*`）下。
- **转换动作**
  - 在字段上添加 `@Autowired`（简单方案）。
  - 选项：支持生成构造函数注入（进阶）。
  - 可选：删除不再需要的 setter。
- **需求点**
  - 规则中可配置：
    - 使用字段注入还是构造注入。
    - 是否自动删除 setter。

---

### 3.6 methodRules：方法级别转换（增 / 删 / 改）

#### 3.6.1 `execute()` / Struts 配置 method → `@RequestMapping`

- **识别条件**
  - 方法名为 `execute`，返回类型为 `String`。
  - 或 `struts.xml` 中 `<action method="foo">` 指定的方法。
- **转换动作**
  - 补充 Spring MVC 映射注解：
    - 结合 `namespace` + `action name` 生成 URL，例如 `/user/list`。
    - 根据规则选择 `@GetMapping` / `@PostMapping` / `@RequestMapping`。
  - 保留返回值 `String`（视图名），或者在 REST 模式下改为 JSON/DTO。
- **需求点**
  - 规则可配置：
    - URL path 模板（如 `"/${namespace}/${actionName}"`）。
    - HTTP 方法默认值与覆盖策略。

#### 3.6.2 返回 `Result`/`String` 的方法映射

- **识别条件**
  - 返回类型为 `com.opensymphony.xwork2.Result`。
  - 或返回 `String` 且依赖 Struts `result` 配置。
- **转换动作（可选）**
  - `Result` → `ModelAndView` 或保持 `String`。
  - 为方法增加必要的 `Model` 参数，并加入 TODO 注释引导人工完成视图映射。
- **需求点**
  - 支持在规则中配置默认映射策略：
    - `Result` → `ModelAndView`。
    - `String` 视图名是否需要变换（去掉前缀/后缀等）。

#### 3.6.3 方法级“增删改”通用规则（包含新增替代方法）

- **核心要求**
  - 对于方法，不仅要支持**修改签名/注解**，还要支持：
    - **删除旧方法**（如只在 Struts 生命周期中有意义的钩子方法、不再需要的辅助方法）；
    - **替换方法**（删除旧方法，生成一个新的 Spring 风格方法，并可选择性迁移原逻辑）；
    - **新增方法**（完全新增一个替代方法或补充方法，可能并非一一对应某个旧 Struts 方法）。

- **典型场景**
  - 删除 `validate()`、某些只依赖 Struts ValueStack 的工具方法，由新的校验机制/工具类替代；
  - 将 `public String execute()` 替换为具名的 `list()` / `save()` 等 REST 风格方法；
  - 为某个迁移后的 Controller 类新增一个全新的 `@GetMapping` 方法，其实现由模板代码自动生成。

- **XML 规则抽象（示意）**

```xml
<MethodRules>
  <!-- 删除某类方法 -->
  <MethodRule enabled="true" priority="100">
    <Name>validate</Name>
    <ReturnType>void</ReturnType>
    <Action>DELETE</Action>
  </MethodRule>

  <!-- 替换某个方法：删除旧方法，新增一个新签名的方法 -->
  <MethodRule enabled="true" priority="120">
    <Name>execute</Name>
    <ReturnType>java.lang.String</ReturnType>
    <Action>REPLACE</Action>
    <NewMethod>
      <Name>list</Name>
      <ReturnType>java.lang.String</ReturnType>
      <Annotations>
        <Annotation>org.springframework.web.bind.annotation.GetMapping</Annotation>
      </Annotations>
      <!-- 可选：方法体模板，允许包含占位符（如 ${className}、${actionName}）-->
      <BodyTemplate>
        <![CDATA[
        // TODO auto-generated by transformer, please refine.
        return "list";
        ]]>
      </BodyTemplate>
    </NewMethod>
  </MethodRule>

  <!-- 纯新增方法：不依赖旧方法存在 -->
  <MethodRule enabled="true" priority="200">
    <Action>ADD</Action>
    <TargetClassPattern>.*UserController</TargetClassPattern>
    <NewMethod>
      <Name>health</Name>
      <ReturnType>java.lang.String</ReturnType>
      <Annotations>
        <Annotation>org.springframework.web.bind.annotation.GetMapping</Annotation>
      </Annotations>
      <BodyTemplate>
        <![CDATA[
        return "OK";
        ]]>
      </BodyTemplate>
    </NewMethod>
  </MethodRule>
</MethodRules>
```

- **需求点**
  - MethodRule 至少要支持三个动作：`ADD` / `DELETE` / `REPLACE`；
  - 新增方法需要能够通过 XML 描述：
    - 方法名、返回类型、参数列表（后续可扩展）、注解列表、访问修饰符；
    - 方法体模板（字符串+占位符），便于在多项目中生成统一风格的骨架代码；
  - 替换方法时，可以选择：
    - 完全舍弃旧方法体，只生成新模板；
    - 或将旧方法体以注释形式拷贝到新方法中，便于人工迁移逻辑。

### 3.7 paramRules：方法参数与 Servlet API 访问

#### 3.6.1 `ServletActionContext` → 方法参数

- **Struts 模式**
  - `ServletActionContext.getRequest() / getResponse() / getSession()`
- **Spring 模式**
  - 通过方法参数注入：
    - `HttpServletRequest request`
    - `HttpServletResponse response`
    - `HttpSession session`
- **转换动作**
  - 查找 `ServletActionContext.getXxx()` 调用：
    - 为方法增加相应参数（若不存在）。
    - 用参数替换原来的静态调用表达式。
  - 自动补充 `jakarta.servlet.http.*` import。
- **需求点**
  - 需避免参数名冲突（简单策略：已有同名则复用）。

#### 3.6.2 请求参数获取 → 注解参数（可选）

- **Struts 模式**
  - `String id = request.getParameter("id");`
- **Spring 模式**
  - 方法参数：`@RequestParam("id") String id`
- **转换动作（可选，风险略大）**
  - 将 `getParameter("xxx")` 转为方法参数 `@RequestParam("xxx")`。
  - 删除/替换对应赋值语句。
- **需求点**
  - 支持在配置中开关该类“积极重构”规则。
  - 默认安全策略：先不删除原代码，只标注 TODO。

### 3.8 routeRules：Struts 配置 → Spring 路由

- **数据来源**
  - `struts.xml`：
    - `<package namespace="...">`
    - `<action name="..." class="..." method="...">`
    - `<result name="success">/WEB-INF/jsp/xxx.jsp</result>`
- **用途**
  - 为 Controller 类/方法生成路径注解：
    - 类级 `@RequestMapping(namespace)`。
    - 方法级 `@GetMapping("/actionName")` 等。
  - 生成注释说明原始 `result` 映射：
    - `// from Struts result 'success' -> /WEB-INF/jsp/xxx.jsp`
- **需求点**
  - 可在规则中定义路径模板与默认 HTTP 方法。
  - 支持根据 result name 生成 TODO 注释或简单策略映射。

## 4. 输出工程结构需求

- **新工程目录结构**
  - `src/main/java`：转换后的 Controller、Service、Interceptor 等。
  - `src/main/resources`：
    - 基础 `application.yml` 或 `application.properties`。
  - 构建文件：
    - `pom.xml`（Maven）或 `build.gradle`（Gradle）。
  - Spring Boot 启动类：
    - `@SpringBootApplication` + `main` 方法。
- **Spring Boot 依赖**
  - 至少包含：
    - `spring-boot-starter-web`
    - （可选）`spring-boot-starter-validation`
    - （可选）`spring-boot-starter-thymeleaf` / JSP 支持（视图层后续处理用）。

## 5. 日志、报告与可视化需求（后续对接 Avalonia UI）

- **日志**
  - 每个文件的转换过程简要记录：
    - 应用的规则（ID/名称）。
    - 成功/失败信息。
    - Warning/TODO（例如复杂拦截器、校验逻辑未自动处理）。
- **报告**
  - 输出一个整体报告（Markdown/HTML/JSON 任选其一或多种）：
    - 统计：
      - 转换的类数量（Action/Interceptor/等）。
      - 应用的关键规则数量。
      - 遗留 TODO 数量与位置。
- **后续对接 Avalonia**
  - 通过命令行参数或 HTTP API 对外暴露：
    - 源目录、规则文件、目标目录。
  - 日志输出到 stdout/stderr，便于 Avalonia UI 实时展示。

## 6. 非目标与限制（当前版本）

- 不处理：
  - Struts 标签库（`<s:xxx>`）和 JSP → Thymeleaf/JSP 的自动转换。
  - 复杂 OGNL 表达式 → Spring EL/Thymeleaf 表达式 的自动迁移。
- 对以下内容仅做“标记/TODO 提示”，不强制自动转换：
  - 自定义复杂拦截器链 → Spring Security/Filter 链。
  - 复杂验证逻辑 → Bean Validation/BindingResult。
  - 与特定框架深度耦合的代码（例如与老版 Spring/其他框架的混用）。
